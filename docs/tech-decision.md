# Nie 技术决策文档

> 版本：v1.0
> 最后更新：2026-04-19

## 核心原则

所有技术决策都围绕一个目标：**快速迭代，验证假设**

---

## 开发流程决策

### 1. 开发执行代理

**决策：** Claude Code 是**唯一**执行代理

**原因：**
- Claude Code 在代码生成和调试方面表现优秀
- 统一执行者，减少沟通成本
- 小马（Hermes）负责规划，Claude Code 负责执行

**规则：**
- 所有代码编写任务都交给 Claude Code
- Claude Code 可以调用终端、文件、浏览器等工具
- 遇到技术难题时，Claude Code 自主解决

---

### 2. 项目管理角色

**决策：** Hermes（小马）负责计划与 review

**小马的职责：**
- ✅ 拆解任务，制定开发计划
- ✅ 给 Claude Code 分配任务
- ✅ 做代码 review
- ✅ 保证架构合理
- ✅ 协调不同模块的集成

**小马不做：**
- ❌ 写核心业务代码
- ❌ 跳过优先级开发
- ❌ 引入复杂系统
- ❌ 直接修改代码（除非是文档类）

---

## AI 模型决策

### 3. 主模型选择

**决策：** 优先使用 **智谱 AI GLM-4V**（主力模型）

**原因：**
- 视觉理解能力强，特别适合摄影分析
- 中文支持优秀，建议更自然
- API 响应速度快
- 技术成熟，稳定性高

**备选方案：**
- 智谱 AI GLM-4V（如果 MiniMax 不满足需求）
- 通义千问 Qwen-VL（作为对比选项）

**行动：**
- 先集成 MiniMax API
- 进行效果测试
- 如果效果不理想，切换到备选方案

**技术实现：**
- 使用 `httpx` 直接调用 HTTP API
- 不安装 Python SDK，减少依赖
- 统一调用方式，方便未来切换模型

---

### 4. 视觉模型 A/B 测试

**决策：** 视觉模型要做 A/B 测试

**测试对象：**
- MiniMax 视觉模型
- 智谱 AI GLM-4V
- 通义千问 Qwen-VL
- 其他国内视觉模型

**测试维度：**
1. **识别准确率**：主体识别、构图分析的准确度
2. **建议质量**：建议的可执行性和实用性
3. **响应速度**：API 调用的延迟
4. **成本**：单次调用的费用
5. **稳定性**：服务的可用性

**测试方法：**
- 准备 100+ 张测试照片（风景、人像、静物、夜景等）
- 每个模型分析相同照片
- 人工评估分析结果
- 记录各项指标
- 综合评分，选择最优

**实施计划：**
- Week 1: 集成 MiniMax，完成基础功能
- Week 2: 添加智谱 AI，开始 A/B 测试
- Week 3: 添加通义千问，扩大测试范围
- Week 4: 根据测试结果，选择主模型

## 移动端技术决策

### 5. 客户端框架选择

**决策：** 先不锁死 Flutter

**原因：**
- iOS 原生开发速度快，用户体验好
- MVP 阶段只需支持 iOS
- Flutter 跨平台优势在单平台场景下不明显
- 需要根据实际开发情况做最终决定

**选项 A：iOS 原生**
- 优点：性能好、开发快、原生体验
- 缺点：后续扩展到 Android 需要重写

**选项 B：Flutter**
- 优点：一套代码，未来扩展容易
- 缺点：相机功能可能不如原生顺滑

**决策标准：**
- Week 1-2: 用 iOS 原生快速验证核心功能
- Week 3: 评估开发速度和体验
- Week 4: 决定是否切换到 Flutter

**如果选择 iOS 原生：**
- 使用 Swift + SwiftUI
- 使用 AVFoundation 处理相机
- 使用 Alamofire 处理网络

**如果选择 Flutter：**
- 使用 camera 插件
- 使用 image_picker 插件
- 使用 dio 处理网络

**当前状态：** 开放选项，等待 Week 3 决策

## 后端技术决策

### 6. 后端框架

**决策：** FastAPI

**原因：**
- 轻量级，快速开发
- 自动生成 API 文档
- Python 生态好，AI 库丰富
- 异步支持，性能优秀

---

### 7. 部署架构

**决策：** 云端优先

**方案：**
- 阿里云 / 腾讯云 / AWS
- Docker 容器化
- Nginx 反向代理
- 对象存储（OSS/S3）

**原因：**
- MVP 阶段快速迭代，不需要考虑本地部署
- 云端扩展性好
- 运维成本低

---

## 架构决策

### 8. 架构风格

**决策：** 分层架构 + 微服务预留

**当前架构（MVP）：**
```
Mobile App → FastAPI → AI Models
```

**未来架构（v2+）：**
```
Mobile App → API Gateway → Image Service
                                → Composition Service
                                → Lighting Service
                                → AI Service
```

**原则：**
- MVP 阶段保持简单
- 代码模块化，方便拆分
- 不引入分布式事务等复杂概念

---

## 技术债务管理

### 9. 何时引入新技术

**决策：** 只有在明确需要时才引入

**不做的：**
- ❌ 不提前引入微服务
- ❌ 不提前引入消息队列
- ❌ 不提前引入缓存
- ❌ 不提前引入监控告警

**做的：**
- ✅ 遇到性能瓶颈时优化
- ✅ 需要解耦时拆分服务
- ✅ 需要监控时添加日志和指标

---

## 开发优先级

### 10. 功能优先级

**决策：** 严格按照优先级开发

1. 构图分析（最高优先级）
2. 构图建议
3. 照片评分
4. 智能裁切
5. 主体识别

**规则：**
- 不跳级开发
- 不并行开发低优先级功能
- 高优先级功能稳定后，再开发下一个

---

## 技术选型原则

### 11. 选型原则

**简单优于复杂：**
- 能用简单方案解决的，不用复杂方案

**成熟优于前沿：**
- 优先选择成熟稳定的技术
- 不做技术实验品

**快速优于完美：**
- MVP 阶段快速验证
- 优化留到 v2+

**实用优于炫技：**
- 不为了用新技术而用新技术
- 每个技术决策都要有明确价值

---

## 决策审查机制

### 12. 决策变更流程

**如果需要变更技术决策：**
1. 小马提出变更建议
2. 说明变更原因和影响
3. 评估变更成本
4. 小马做最终决策
5. 更新本文档
6. 通知团队成员

**示例：**
- 如果 MiniMax 效果不好 → 切换到智谱 AI
- 如果 iOS 原生开发太慢 → 切换到 Flutter
- 如果 FastAPI 性能不够 → 切换到 Go
