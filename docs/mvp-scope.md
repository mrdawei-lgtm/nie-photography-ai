# Nie MVP 范围定义

> 版本：v1.0
> 最后更新：2026-04-19

## MVP 核心目标

验证假设：**AI 能否帮助用户拍出更好的照片？**

---

## MVP 功能范围

### 输入

**输入一张照片**
- 用户拍照或从相册选择
- 支持常见格式：JPG, PNG, HEIC
- 图片大小限制：≤ 10MB

---

### 输出

#### 1. 结构化分析

返回以下结构化数据：

**主体信息：**
- 主体类型（人物、动物、物体、风景）
- 主体位置（归一化坐标）
- 主体清晰度

**构图信息：**
- 构图评分（0-100）
- 使用的构图规则
- 违反的构图规则

**光线信息：**
- 光线类型（顺光、侧光、逆光）
- 曝光评估（过曝、正常、欠曝）
- 亮度值（0-1）

**整体评分：**
- 综合评分（0-100）
- 主要问题列表

#### 2. 构图建议

给出 2-3 条可执行的构图建议：

示例：
- "向左移动一点"
- "把主体放到右侧三分之一"
- "靠近一点"

**要求：**
- 每条建议都必须可执行
- 建议要简洁明了
- 不讲构图理论

#### 3. 光线建议

给出 1-2 条光线相关建议：

示例：
- "光线在背后，建议转向"
- "画面偏暗，建议提高曝光或换角度"

#### 4. 主体建议

给出 1-2 条主体相关建议：

示例：
- "主体不突出，建议靠近"
- "背景干扰较多，建议换个角度"

#### 5. 可选生成裁切方案（实验性功能）

**状态：** 可选，非必须

如果时间允许，实现：
- 自动计算最优裁切区域
- 在原图上标注建议裁切框
- 用户点击确认后生成裁切后的图片

**注意：** MVP 阶段这个功能可以不做，或作为隐藏功能

---

## MVP 不包含的功能

### ❌ 明确不做

#### 1. 实时分析
- 不做相机预览时的实时分析
- 只做拍后分析

**原因：** 实时分析技术复杂，MVP 阶段先验证核心价值

#### 2. 自动抓拍
- 不做自动识别最佳瞬间并拍摄
- 用户需要手动拍照

**原因：** 自动抓拍技术难度高，且用户接受度未知

#### 3. 视频分析
- 不支持视频分析
- 只支持静态图片

**原因：** MVP 验证的是拍照场景

#### 4. 修图 / 滤镜
- 不提供任何修图功能
- 不提供滤镜

**原因：** 不是我们的定位，我们只做"拍摄助手"

#### 5. 社交系统
- 不做用户系统
- 不做社交分享
- 不做内容社区

**原因：** 不在 MVP 范围内

#### 6. 摄影教学
- 不做摄影教程
- 不做知识库
- 不做技能测试

**原因：** 教学是副产物，不是主目标

## MVP 技术范围

### 必须实现

#### 后端
- ✅ 图片上传接口
- ✅ 图片分析接口
- ✅ 返回结构化数据
- ✅ 集成 AI 模型（MiniMax / 智谱 AI）
- ✅ 基础错误处理

#### 移动端
- ✅ 相机拍照功能
- ✅ 相册选择功能
- ✅ 图片上传到后端
- ✅ 展示分析结果
- ✅ 展示建议列表

### 可以简化

#### 构图规则
- MVP 只实现三分法
- 居中构图可以作为简单规则

#### 主体识别
- 简化为"主体位置检测"
- 不做细粒度分类

#### 裁切方案
- 只做标注，不生成新图
- 或者完全不做

---

## MVP 成功标准

### 用户层面

1. **用户能拍出更好的照片**
   - A/B 测试：使用 Nie vs 不使用 Nie
   - 评分提升 > 20%

2. **用户能理解 AI 的建议**
   - 用户调研：建议理解度 > 80%
   - 用户能按照建议操作

3. **用户愿意再次使用**
   - 留存率：次日留存 > 50%
   - 周留存 > 30%

### 技术层面

1. **API 响应时间**
   - 平均响应时间 < 5 秒
   - 95% 请求 < 10 秒

2. **识别准确率**
   - 主体识别准确率 > 80%
   - 构图分析准确率 > 70%

3. **建议质量**
   - 建议可执行率 > 90%
   - 建议有用率 > 70%

## MVP 开发周期

### 时间安排

**Week 1: 后端基础**
- 搭建 FastAPI 项目
- 集成 MiniMax API
- 实现图片分析接口
- 测试和调试

**Week 2: 构图分析引擎**
- 实现三分法规则
- 实现构图评分算法
- 优化 AI 提示词
- 测试和调试

**Week 3: 移动端开发**
- 创建 iOS 项目
- 实现相机功能
- 实现图片上传
- 展示分析结果

**Week 4: 集成测试和优化**
- 端到端测试
- Bug 修复
- 性能优化
- 准备内测

---

## MVP 验证计划

### 内测阶段（Week 5-6）

1. **招募内测用户**
   - 50-100 名用户
   - 覆盖不同年龄、摄影水平

2. **收集反馈**
   - 用户满意度调查
   - 建议有用性评估
   - 使用场景分析

3. **数据分析**
   - 留存率分析
   - 使用频次分析
   - 功能使用率分析

### 决策点（Week 6 末）

**如果成功：**
- 继续迭代 v1.1
- 增加更多构图规则
- 优化建议质量

**如果不成功：**
- 分析失败原因
- 调整产品方向
- 可能需要重新定位

## MVP 风险和应对

### 风险 1：AI 分析不准确

**应对：**
- A/B 测试多个模型
- 收集用户反馈
- 持续优化提示词

### 风险 2：建议不够实用

**应对：**
- 用户访谈
- 快速迭代建议生成逻辑
- 基于用户行为数据优化

### 风险 3：用户不信任 AI

**应对：**
- 展示分析过程
- 提供示例照片对比
- 强调 AI 是辅助工具

### 风险 4：响应时间太慢

**应对：**
- 优化图片压缩
- 使用更快的 API
- 考虑边缘计算

## MVP 验收标准

### 功能验收

- [ ] 用户能拍照并上传
- [ ] 后端能分析图片并返回结果
- [ ] 移动端能正确展示分析结果
- [ ] 建议清晰、可执行

### 性能验收

- [ ] API 响应时间 < 5 秒（平均）
- [ ] App 启动时间 < 3 秒
- [ ] 拍照到展示结果 < 10 秒

### 质量验收

- [ ] 主体识别准确率 > 80%
- [ ] 构图分析准确率 > 70%
- [ ] 建议 90% 可执行
- [ ] 建议 70% 有用

### 用户体验验收

- [ ] 界面简洁易懂
- [ ] 操作流程顺畅
- [ ] 错误提示友好
- [ ] 整体体验流畅
